# Agent 基本设计原理

本文总结本项目最核心的设计原则，帮助你快速理解这个 toy agent 为什么这样实现。

## 1. 最小可用闭环优先

系统只保留一个最小但完整的执行回路：

1. 读取任务与历史状态
2. 调用模型产出下一步动作
3. 执行本地工具
4. 将结果回填到历史
5. 循环直到 finish 或步数上限

这个闭环在教学和调试上可解释性强，便于定位每一步的输入输出。

## 2. 决策协议显式化（JSON Action Protocol）

模型不能自由输出任意文本，而是必须返回一个 JSON 动作对象。
动作空间被限制为少量原子操作：

- read_file
- search_text
- write_file
- run_command
- finish

这样做的目的：

- 降低模型行为不确定性
- 让 agent 控制流可验证
- 便于在未来扩展动作而不破坏主循环

## 3. 关注点分离

代码按职责拆分为四层：

- main.py: 启动、参数和环境配置
- llm.py: 模型请求与响应解析边界
- tools.py: 本地工具实现与安全约束
- agent.py: observe-decide-act 主循环

每层尽量只做一件事，便于独立演进和替换。

## 4. 工具是能力边界

agent 能做什么，不由模型自由决定，而由 TOOLS 映射决定。
通过限制工具集合，可以把能力边界收敛到可控范围内，避免隐式副作用。

## 5. 工作区隔离与路径安全

所有文件访问都经过工作区路径校验，拒绝越界路径。
这是最基础的本地安全防线，避免模型读写工作区之外的数据。

## 6. 外部反馈驱动迭代

run_command 提供真实环境反馈（例如测试是否通过）。
agent 不依赖“自我感觉正确”，而是根据命令输出继续修正或结束。

## 7. 上下文预算控制

文件和命令输出会做截断，防止把过长文本回灌给模型，导致上下文污染或成本上升。
这是面向长期运行稳定性的基础策略。

## 8. 失败优先可观测

- 模型请求失败会抛出明确错误
- 非法动作会返回结构化错误
- 每一步决策和工具结果都会打印

目标是让问题可以被快速看见、快速定位。

## 9. 教学优先于工程完备

本项目有意保持简单，刻意不做以下复杂特性：

- diff 级补丁编辑
- 长期记忆
- 流式 token 输出
- 命令沙箱与细粒度权限模型

这样可以把注意力集中在 agent 的核心机制，而不是工程细节。

## 10. 可演进方向

在不破坏上述原则的前提下，可逐步增强：

- 将 write_file 升级为补丁式编辑
- 引入更细粒度的工具权限控制
- 将历史压缩为摘要，提升长任务稳定性
- 增加结构化评估与回归测试
