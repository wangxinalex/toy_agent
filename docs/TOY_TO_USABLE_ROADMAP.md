# 从 Toy 到可用：后续演进路线

这份文档规划 V3 及之后要解决的问题。每一步都只引入最少的变化，保持独立可验证。

## 已完成的版本

| 版本 | 解决的问题 | 为什么重要 |
|------|-----------|-----------|
| V1 单轮对话 | agent 的最小闭环能跑起来 | 奠基。不理解 V1 就看不懂后续 |
| V2 多轮交互 | 进程内持续对话 | 体验从"命令行工具"变为"对话助手" |

## V3：会话持久化

**要解决的问题**：关掉进程，上下文全部丢失。

**核心思路**：
- 在 `sessions/` 目录下按时间戳保存 JSON 格式的历史文件。
- CLI 新增 `--resume <session-id>` 恢复上次会话。
- 对 history 做长度截断，避免无限增长撑爆上下文窗口。

**为什么优先做这个**：持久化是多轮交互的前提。没有它，你每次都要从头开始解释背景，V2 的实际可用性大打折扣。

## V4：补丁式编辑

**要解决的问题**：当前 `write_file` 是整文件覆写。改一行代码也要生成完整文件，浪费 token、容易出错。

**核心思路**：
- 新增 `apply_patch` 工具，接受 `path` + `old_snippet` + `new_snippet`，做精确的文本替换。
- system prompt 里约束模型：默认用补丁编辑，只有创建新文件时才用 `write_file`。
- 补丁应用失败时返回明确错误，让模型重试。

**为什么 V4 在 V3 之后**：补丁编辑不依赖持久化，但持久化对用户体验的提升更大、实现成本更低。

## V5：工具权限与命令白名单

**要解决的问题**：`run_command` 使用 `shell=True`，任何命令都能执行。

**核心思路**：
- 建立安全命令白名单（如 `pytest`、`ruff`、`python -m`、`git status`）。
- 禁止危险命令：删除、网络下载、权限升级、fork 炸弹等。
- 命令执行加超时（如 30 秒）和输出长度限制。

**安全警示**：这个 agent 目前是本地学习工具。在没有完善 V5 之前，不要在不信任的环境里运行。

## V6：自动验证闭环

**要解决的问题**：agent 改完代码后不会自动验证，不知道改得对不对。

**核心思路**：
- 在 system prompt 层面约定：修改代码后必须至少运行一个验证命令（如 `pytest`、`python -m compileall`）。
- 验证失败则 agent 需要据错误信息修改代码，重新验证，直到通过才 finish。
- finish 时附带验证证据（测试通过或编译成功）。

**为什么这个最后做**：自动验证依赖前面的基础——你需要持久化来跨 session 追踪验证状态，需要安全的命令执行环境来跑验证。

## 学习建议

每个版本都可以对照着上一个版本的代码读：
- **增量很小**：每次改动通常只涉及 1-2 个文件。
- **目的明确**：每个版本解决一个具体问题。
- **独立可跑**：版本目录里的代码永远是完整、可执行的。
